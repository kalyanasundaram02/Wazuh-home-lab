# Suricata IDS Integration

## Why Suricata, alongside Wazuh

Wazuh's native rules in this lab are host-based — they read log files (`auth.log`) that only exist because *something on the host* (like `sshd`) chose to log an event. That's powerful, but it means Wazuh alone can't see things happening at the network/packet level that never generate an application log line — like a port scan that only touches closed ports, or a probe against a service that isn't even running.

Suricata fills that gap: it inspects traffic on the wire directly, and its alerts (`eve.json`) are ingested into Wazuh as an additional log source, so both host-based and network-based detections show up in the same dashboard.

## Configuration

Full file: [`configs/suricata-network-config.yaml`](../configs/suricata-network-config.yaml)

Key setting:
```yaml
vars:
  address-groups:
    HOME_NET: "[192.168.0.100]"
    EXTERNAL_NET: "any"
```

`HOME_NET` tells Suricata which IP(s) count as "ours" — traffic to/from this range gets evaluated against the full rule set as protected assets. Right now it's scoped to just the manager itself (`192.168.0.100`).

**Improvement worth noting:** widening `HOME_NET` to the full lab subnet (e.g. `192.168.0.0/24`) would let Suricata also flag suspicious traffic *between* other hosts in the network (like the Windows host or future additional agents), not just traffic aimed directly at the manager. Right now, an attack against another machine on the same network wouldn't be seen by Suricata at all.

Suricata's alert output is ingested by Wazuh via this `ossec.conf` block (see [Wazuh Server Setup](2.wazuh-server-setup.md)):
```xml
<localfile>
  <log_format>json</log_format>
  <location>/var/log/suricata/eve.json</location>
</localfile>
```

## What it caught

During the [reconnaissance phase](3.network-reconnaissance.md), Suricata flagged a burst of scanning-related alerts around the same time the Nmap scans were run:

![Wazuh dashboard - Suricata port scan alerts](../evidence/screenshots/2.png)

| Signature | What it means |
|---|---|
| ET SCAN Potential SSH Scan | Probing traffic toward the SSH port, consistent with the recon phase |
| ET SCAN Suspicious inbound to Oracle SQL port 1521 | Port sweep touched a database service port, even though nothing is listening there |
| ET SCAN Suspicious inbound to PostgreSQL port 5432 | Same pattern, different DB service |
| ET SCAN Suspicious inbound to MSSQL port 1433 | Same pattern, MSSQL — full sample alert saved below |
| ET SCAN Suspicious inbound to MySQL port 3306 | Same pattern, MySQL |
| ET INFO Possible Kali Linux hostname in DHCP Request | Suricata fingerprinted the attacker's OS from DHCP traffic — useful attribution signal |
| Listened ports status changed (rule 533) | Host-based complement: Wazuh's own `netstat`-based check noticed a port state change on the manager itself |

This is a good example of network-based detection catching something host-based logging never would: none of those database ports had a service running behind them, so there was no application log to generate an `auth.log`-style Wazuh alert — but Suricata still saw the probe traffic hit the wire.

## Sample alert (MSSQL scan)

Full JSON: [`evidence/alerts/alert-suricata-portscan-mssql.json`](../evidence/alerts/alert-suricata-portscan-mssql.json)

```
Signature: ET SCAN Suspicious inbound to MSSQL port 1433
Severity: Informational (2)
Source: 192.168.0.7 → Destination: 192.168.0.100:1433
Wazuh rule: 86601, level 3
```

Note the low Wazuh level (3) here compared to the SSH brute-force alerts (up to 12) — this reflects that a single scan touch on a closed/unused port is low-signal on its own. It only becomes meaningful in the context of *many* such alerts arriving together across multiple ports in a short window, which is exactly the pattern seen here — a good discussion point for how alert volume/pattern matters as much as any single alert's severity.
