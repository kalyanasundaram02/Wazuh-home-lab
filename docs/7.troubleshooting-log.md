# Troubleshooting Log

Real problems hit while building this lab, documented as Problem → Root Cause → Fix → Takeaway. This is deliberately kept in the repo — the debugging process is as much a part of the skill set as the attack/detection work itself.

---

## 1. VS Code Remote-SSH stalling over Bridged networking

**Problem:** Connecting to the Ubuntu Server VM via VS Code's Remote-SSH extension consistently stalled while downloading the VS Code Server component onto the remote host. A plain terminal SSH session worked fine — so this wasn't a basic connectivity failure.

**Investigation:**
- Ruled out the OS install itself by reinstalling Ubuntu Server clean (this is also where the version was standardized from 26.04 down to 24.04 LTS, after hitting the same slowdown on both — see note below).
- Confirmed SSH itself was healthy: service running and enabled, static IP configured, gateway/DNS resolving correctly, and a direct terminal SSH session connected without issue.
- Manually installed the VS Code Server files to rule out the extension's own install logic as the problem — same stall.
- Used `iperf3` to measure raw throughput between Windows and the Ubuntu VM directly: LAN throughput was fine, but internet-bound downloads (which is what fetching the VS Code Server needs) were crawling around **65 kbps** over the VM's Bridged network adapter.
- Switched the VM to NAT networking temporarily as a comparison test — download speed returned to normal immediately, isolating the issue specifically to Bridged mode, not Ubuntu, not SSH, and not VS Code's install logic.

**Root cause:** The VirtualBox network adapter type in use (Intel PRO/1000 MT Desktop) was bottlenecking throughput specifically under Bridged mode — an emulation overhead issue between the virtual NIC and the host's physical network interface.

**Fix:** Changed the VM's network adapter type from Intel PRO/1000 MT Desktop to **Paravirtualized Network (virtio-net)**. Download speed jumped from ~65 kbps to ~10 MB/s immediately, and Remote-SSH connected and downloaded the VS Code Server normally afterward.

**Takeaway:** When a VM's network is technically "up" but strangely slow, don't default to assuming it's a DNS/routing/firewall issue — the *virtual NIC emulation type itself* can be the bottleneck, and it can be mode-specific (it only showed up in Bridged here, not NAT). Comparing throughput across adapter types early on would have shortened this significantly.

## 2. Ubuntu Server version standardization (26.04 → 24.04)

**Problem:** Initially installed Ubuntu Server 26.04 for the manager VM, but hit the exact same Bridged-networking slowdown described above on that version too.

**Fix:** Standardized on Ubuntu Server 24.04 LTS instead — matching the version used in the institute's own lab environment, which also ruled out any version-specific quirk as a factor once the adapter-type fix (above) was applied.

**Takeaway:** Matching your lab's OS version to a known-working reference environment (in this case, the institute's lab) removes one variable when troubleshooting — useful when you're debugging infrastructure and don't want the OS version itself to be an open question.
