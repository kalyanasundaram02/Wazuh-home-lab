# Incident Report: SQL Injection Against DVWA (UNION-Based Enumeration)

## 1. Scenario

**Date:** 2026-07-06
**Target:** DVWA (Damn Vulnerable Web Application), hosted on Ubuntu Server `192.168.0.100`
**Attacker source:** `192.168.0.4` (Kali Linux VM)
**Vulnerable endpoint:** `/DVWA/vulnerabilities/sqli/`
**Technique:** UNION-based SQL Injection (manual, via Burp Suite Repeater)

The DVWA SQL Injection module accepts a `User ID` parameter and reflects the result directly into the page without sanitizing input. This was exploited to enumerate database schema information rather than just bypassing the login logic.

## 2. Attack Details

**Injected payload:**
```
1' UNION SELECT table_name, table_schema FROM information_schema.tables WHERE table_schema=database()-- -
```

**What this does:**
- `1'` closes out the original quoted string in the vulnerable query
- `UNION SELECT table_name, table_schema` appends a second query that pulls table names and their schema instead of the expected user record
- `FROM information_schema.tables WHERE table_schema=database()` restricts results to tables belonging to the current database (rather than every database on the server)
- `-- -` comments out the rest of the original SQL statement so the syntax remains valid

**Result:** the application returned every table name in the current database, effectively mapping out the schema:

| Table Name |
|---|
| guestbook |
| security_log |
| access_log |
| users |

This is a classic **information disclosure via SQLi** — the attacker now knows table names to target for follow-up injection (e.g. pulling credentials from `users`).

## 3. Detection

This attack was **not** caught by a custom Wazuh rule — it was flagged by **Suricata's existing ET (Emerging Threats) open ruleset**, which Wazuh ingests via `eve.json`.

| Field | Value |
|---|---|
| Detection engine | Suricata IDS |
| Signature ID | 2016672 |
| Signature | `ET WEB_SERVER SQL Errors in HTTP 200 Response (error in your SQL syntax)` |
| Wazuh Rule ID | 86601 |
| Wazuh Rule Level | 3 |
| Severity (Suricata) | Minor (2) |
| Category | Potentially Bad Traffic |
| Times fired | 17 |

**Important note for future readers of this repo:** this was signature-based detection from a pre-existing community ruleset, not a rule I authored. The signature happens to trigger on MySQL error strings appearing in HTTP 200 responses — meaning it caught this because the injected query produced a SQL syntax error message somewhere in the exchange, not because it specifically understood "this is a UNION-based enumeration attempt." A more precise detection would require a custom rule tuned to DVWA's specific injection patterns (covered separately).

## 6. Analysis

- The alert level (3) is low, and the signature is generic — it would fire on *any* SQL error appearing in an HTTP 200 response, not specifically on UNION-based enumeration
- As-is, this alert alone would not tell a Tier 1 analyst that schema enumeration occurred — only that "a SQL error was present." Determining attacker intent required manually reviewing the `url` and `http_refer` fields to see the actual injected payload
- This highlights a real detection gap: **the current setup detects that something went wrong on the SQL side, but doesn't classify the technique or severity accurately**

## 7. Recommendation

- Tune or supplement the existing Suricata signature with a rule that specifically flags SQL keywords in URL parameters (`UNION`, `SELECT`, `information_schema`, `--`) rather than relying only on error-message detection
- Increase the Wazuh rule level for this signature given the target is a known vulnerable web app in a monitored network segment — level 3 under-represents the risk
- Correlate this alert with source IP reputation / frequency (repeated SQLi attempts from the same IP in a short window should escalate severity, similar to brute-force frequency rules)
- Recommend WAF-layer input validation in front of the application in a production scenario, since detection alone doesn't prevent data exposure
